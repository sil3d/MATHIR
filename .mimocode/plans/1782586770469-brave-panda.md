# MATHIR v9.0 « Universal » — Plan d'unification

## Objectif
Un seul système cohérent qui rend MATHIR universel : IDE-agnostique (opencode/claude/cursor/windsurf/CLI), un seul schéma DB, une seule formule de decay, un seul arbre de modules, un seul transport, auth opt-in non-cassante. Résout les 5 loose ends design restants.

## Principe directeur
**Loopback = trusted (zéro changement pour l'usage local type mimocode). Non-loopback = requiert un token.** Toute l'unification respecte cette règle pour ne pas casser le redémarrage.

---

## Les 6 piliers

### Pilier 1 — Arbre de modules unique (tuer la drift brain/)
Rien n'importe le namespace `brain` (vérifié). Action : absorber `brain/` dans `mathir_lib/`, verdict par fichier (issu du recon) :
- **mathir_brain.py** → garder la version `brain/` (portable : `_BRAIN_DIR`/env `MATHIR_PORT`) + corriger le ping `\n` framing pour matcher le daemon.
- **mathir_watchdog.py** → **fusion** : backoff + PID lockfile (brain/) **+** flag `--kill` (mathir_lib/). Aucune des deux n'est un sur-ensemble.
- **mathir_inject_proxy.py** → garder la version `mathir_lib/` (a la sanitization anti prompt-injection ; brain/ est une régression sécurité).
- Retirer `"brain"` de `pyproject.toml [tool.setuptools] packages` et `package-dir`. Supprimer `mathir_mcp/brain/` (garder un README de redirection).
- **Fix bonus** : `mathir-mcp` console_script cassé (le module `mathir_mcp_server.py` n'a pas de `main()`) — ajouter `def main(): mcp.run()`.

### Pilier 2 — Un seul schéma + migration one-shot
Le schéma « new » devient le SEUL. Migration des legacy DBs existantes :
- Créer `mathir_mcp/mathir_lib/mathir_migrate.py` : lit `LEGACY_DB_PATH`, applique le delta SQL exact (table de mapping issue du recon : `modality_text`→`content` rename, extraction `agent/block_type/label/priority/project` depuis `json_extract(metadata,...)`, conversion `timestamp` REAL→`created_at` ISO via `datetime(ts,'unixepoch')`, fold `recall_count` dans metadata JSON, drop `stability/embedding/embedding_dim/modality/provider/model`), backup original en `.legacy.bak`.
- Supprimer la branch `_schema_kind()=="legacy"` dans `mathir_vec.py` (store/search/touch_recall/decay) — l'agent a déjà ajouté les guards today ; étape finale = supprimer les paths legacy.
- Supprimer `mathir_daemon.py` (TCP raw, déjà « retired » en v8.5.0 mais encore lancé par brain) — le HTTP server (7338) est le seul serveur.
- Nettoyer `mathir_consolidate.py` / `mathir_spread.py` : leurs guards today rendent les DB new-schema gracieux ; v9 = supprimer les paths legacy-only.

### Pilier 3 — Un seul transport (HTTP) + auth opt-in
- **Un serveur** : HTTP unifié Flask+Waitress (7338). Le stdio MCP (`mathir-mcp`) reste un thin proxy vers lui.
- **Auth non-cassante** :
  - Bind loopback (`127.0.0.1`/`::1`, défaut) : **pas d'auth** → mimocode, MCP stdio, CLI local inchangés.
  - Bind non-loopback (`--host 0.0.0.0` ou LAN) : **refuse de démarrer** sauf si `MATHIR_AUTH_TOKEN` set ; tout `/api/*` vérifie `Authorization: Bearer <token>`.
- Dashboard (`mathir_stats_server`) : même règle. CORS déjà loopback-only (fixé today).
- Supprimer `mathir_daemon.py` (raw TCP, zero auth, pose problème de framing) — retire le risque.

### Pilier 4 — Une seule formule de decay
- **Canon** : la formule linéaire Ebbinghaus de `mathir_vec.decay_all` : `stability = max(0, stability - days_since * decay_rate / 30)`.
- `bump_recall` : additif `+0.1` plafonné à 1.0 (déjà aligné today sur dropin ; vérifier alignement mathir_vec).
- Supprimer `mathir_consolidate.apply_decay` (×0.95) — `consolidate.py` délègue à `VecMemory.decay_all`.
- **Point d'entrée unique** : `VecMemory.run_maintenance()` = decay + promote + dedupe + build_links. Le brain/cron l'appelle ; plus jamais 3 formules concurrentes.

### Pilier 5 — Packaging IDE-agnostique + paths universels
- `mathir_paths.py` centralise déjà (fixé today, back-compat `~/.config/opencode` → `~/.config/mathir`).
- **Fix la seconde résolution** : `mathir_lib/__init__.py get_db_path()` utilise `~/.local/share/mathir/` (XDG) — diverge de `mathir_paths.PROJECTS_DIR`. Unifier sur `mathir_paths`. **Une seule résolution.**
- `pyproject.toml` expose un console_script par rôle :
  - `mathir-mcp` (stdio MCP, pour opencode/claude/cursor)
  - `mathir-server` (HTTP unifié)
  - `mathir-client` (CLI HTTP)
  - `mathir-dashboard`
  - `mathir-migrate` (one-shot)
  - `mathir-brain` (orchestrateur)
- README : **1 ligne de config par client IDE** (toutes = `mathir-mcp` comme commande MCP). Zéro mention de chemin IDE dans le code.

### Pilier 6 — Tests + vérification
- Nouveaux tests dans `mathir_mcp/tests/` :
  - `test_migration.py` : legacy→new round-trip (row counts, content equivalence, metadata JSON shape).
  - `test_auth.py` : loopback passe sans token ; non-loopback refuse sans token ; non-loopback + token passe.
  - `test_decay.py` : une seule formule ; `run_maintenance()` idempotent.
  - `test_module_tree.py` : `import mathir_mcp.mathir_lib.*` marche ; `brain` namespace n'existe plus.
- Commande de test : `pytest mathir_mcp/tests/ mathir_dropin/tests/`.
- Smoke : `python -m mathir_mcp` démarre ; `mathir-client ping` répond ; `mathir-migrate --dry-run` liste les DBs à migrer.

## Séquence d'exécution (l'ordre compte)

1. **Migration tools d'abord** (non-cassant) : `mathir_migrate.py` + tests migration. Tourne `--dry-run` sur cette machine pour valider le delta SQL sans toucher aux données.
2. **Unification paths** : unifier `__init__.get_db_path` sur `mathir_paths`. (back-compat préservée.)
3. **Pilier 1 modules** : absorber brain/, fix console_script, supprimer `brain` package.
4. **Pilier 4 decay** : `run_maintenance()` + suppression `apply_decay` concurrent.
5. **Pilier 2 schema** : appliquer la migration sur les legacy DBs de la machine, puis supprimer les paths legacy dans mathir_vec/consolidate/spread.
6. **Pilier 3 transport/auth** : supprimer `mathir_daemon.py`, ajouter l'auth opt-in au HTTP server + dashboard.
7. **Pilier 5 packaging** : pyproject console_scripts + README multi-IDE.
8. **Pilier 6 tests** : ajouter les 4 suites, vérifier end-to-end.

Chaque étape est indépendante et committable séparément. Si une étape casse, on revert juste cette étape.
